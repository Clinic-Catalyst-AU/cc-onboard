How to Drive Your Claude

Your whole system, one page. Open Terminal, type claude, then talk to it like a very capable new team member.

The BasicsDriving 101

  • Start: open Terminal (Cmd+Space, type "Terminal"), then type claude and press enter.
  • Talk normally. "Write me three Instagram captions about winter skin" works. You don't need commands for everything - the slash commands below are shortcuts to the big machines.
  • It asks before it acts. When Claude wants to run something, it shows you first - press enter/y to approve. Nothing sneaky happens without you seeing it.
  • Press Esc to interrupt if it's going the wrong way. Just tell it what you want instead.
  • Paste errors. If anything breaks, copy the error, paste it in, say "fix this". It usually can.
  • Where things live: client folders are in ~/Clients (shared with Kelly - same files, both machines). Content lands in the client folder or the Content-Library.

Making ChangesHow to Edit Things

You have full edit rights across the whole system - same as Kelly. You never edit code or HTML by hand: you tell Claude what to change and it does the work, shows you, and you approve.

  • Client docs & content (brand guides, offers, captions, anything in the client folders): just ask. "Open Amar Skin Clinic's brand guide and soften the tone of the voice section" - Claude edits the file, shows you exactly what changed, you approve. Because the folders are shared, Kelly sees it too.
  • The live workshop pages (itinerary, workbook, guides - anything on the toolkit site): tell Claude the page and the change, and to deploy it. "On the workshop toolkit itinerary page, change morning tea to 10:30. Clone the repo fresh, make the edit, push it live, and verify the live page shows the change before telling me it's done." That last part is a house rule - never trust "pushed", only "verified live".
  • The skills themselves: if a skill keeps getting something wrong, you can fix it for both of you. "The cc-cover skill always makes headlines too small - update the skill and push it so Kelly gets the fix too."
  • Getting Kelly's updates: when she ships improvements, one line brings them to your machine: cd ~/Systems/cc-aios && git pull && bin/apply-to-machine.sh
  • Day-to-day rhythm: one job per conversation works best. Be specific about the outcome you want, not the steps. Claude already knows the CC rules (AHPRA, drafts-only, no fabrication) - they're baked into its system.

House RulesThe Non-Negotiables

  • Everything social lands in GHL as a DRAFT. A human approves in the planner - nothing auto-publishes. Ever.
  • AHPRA is a hard gate. The skills check claims automatically and will stop and tell you rather than ship something non-compliant. If it stops, that's it working.
  • Never fabricate. No invented testimonials, stats or results - the skills use verbatim quotes from real sources only.
  • Big image-generation batches: check with Kelly first (shared key for now).
  • Client work saves into the shared folders - so Kelly sees what you build and you see hers, automatically.
Stuck? First: just tell Claude what's wrong - it fixes most things itself. Still stuck: paste it into the Install Helpline chat or text Kelly.

Wrapping UpEnd every session with /cc-shutdown

When you're done for the day, type /cc-shutdown (or just say "wrap up"). It reports everything you touched as SHIPPED / BLOCKED (with the exact next action) / HALF-DONE, pushes any skill changes so Kelly's machines get them, saves a memory of where things stand, and tells you the first thing to pick up next session. Two minutes, zero silent loose ends.

What it does: downloads whatever Kelly has improved in the shared system and installs it on this machine. When: start of every work session (10 seconds), and any time Kelly says "I've shipped something".
